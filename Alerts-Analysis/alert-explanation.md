## 1. Executive Summary

This document presents a complete analysis of security alerts generated by **Wazuh SIEM** during two simulated cyber attacks performed in a controlled home lab environment.

### 🔴 What Happened:

Two attacks were simulated from **Kali Linux (Attacker VM)** targeting **Windows Host (Victim Machine)**:

| # | Attack | Tool Used | Result |
|---|--------|-----------|--------|
| 1 | Network Port Scan | Nmap | Open ports discovered — SMB (445), RDP (3389) exposed |
| 2 | SMB Brute Force | Hydra | Multiple login failures detected — 23 alerts generated |

| Metric | Value |
|--------|-------|
| **Total Alerts Generated** | 23+ |
| **Critical Alerts (Level 10+)** | 2 |
| **High Alerts (Level 5-9)** | 21+ |
| **Attacks Detected** | 2 out of 2 (100% detection rate) |
| **Average Detection Time** | < 5 seconds |
| **Agent Reporting** | Window-agent (Windows Host) |

### ✅ Conclusion:

Wazuh successfully detected **all simulated attacks** in real-time, demonstrating its effectiveness as a SIEM solution for small-to-enterprise environments.

| Field | Value |
|-------|-------|
| **Attack Type** | Network Port Scan |
| **Tool** | Nmap |
| **Source** | Kali Linux VM |
| **Target** | Windows Host (192.168.x.x) |

| **Command Used** | `sudo nmap -A <Windows-IP>` |

### What Nmap Discovered on Victim:

| Port | State | Service | Risk |
|------|-------|---------|------|
| **135/tcp** | open | msrpc | Medium |
| **139/tcp** | open | netbios-ssn | Medium |
| **445/tcp** | open | microsoft-ds (SMB) | 🔴 HIGH |
| **3389/tcp** | open | ms-wbt-server (RDP) | 🔴 HIGH |
| **49668/tcp** | open | msrpc | Low |

### Critical Finding:
```
⚠️  PORT 445 (SMB) IS OPEN
This directly led to Attack 2 — SMB Brute Force
An attacker always scans first, then attacks the open service.
This is the real-world attack chain demonstrated in this lab.
```

### Wazuh Detection for Port Scan:

> **Note:** Wazuh's detection of port scans depends on Windows Firewall logging. If Windows Firewall logs inbound connection attempts, Wazuh picks them up via Event ID 5156 (Filtering Platform Connection).

| Event ID | Description |
|----------|-------------|
| **5156** | Windows Filtering Platform allowed connection |
| **5152** | Windows Filtering Platform blocked a packet |

## 6. Rule Analysis — Deep Dive

### Rule 60122 — Logon Failure

```
Rule ID      :  60122
Level        :  5 (Medium)
Description  :  Logon Failure — Unknown user or bad password
Trigger      :  Single Windows Event ID 4625
Log Source   :  Windows Security Event Log
Category     :  Authentication Failure
```

**What triggers it:**
- Any single failed login attempt on Windows
- Windows Event ID 4625 is generated
- Wazuh agent forwards it to Wazuh Manager
- Rule 60122 matches the pattern and creates alert

**Why Level 5:**
- A single failed login could be a user mistyping their password
- Not immediately critical — but should be monitored

---

### Rule 60204 — Multiple Windows Logon Failures

```
Rule ID      :  60204
Level        :  10 (High)
Description  :  Multiple Windows Logon Failures
Trigger      :  Multiple Rule 60122 alerts in short timeframe
Log Source   :  Correlated from Windows Security Events
Category     :  Brute Force Attack Detected
Frequency    :  8+ failures within 120 seconds from same source
```

**What triggers it:**
- Wazuh sees many Rule 60122 alerts in a short period
- Correlation engine recognizes brute force pattern
- Escalates severity from Level 5 → Level 10
- This is Wazuh's **threat correlation** in action

**Why Level 10:**
- Multiple failures = deliberate attack, not human mistake
- High probability of automated tool being used
- Immediate SOC analyst attention required

**Rule Hierarchy:**

```
Windows Event ID 4625 (Single Login Failure)
         ↓
Wazuh Rule 60122 — Level 5 (Individual Alert)
         ↓
[Multiple 60122 alerts in short time]
         ↓
Wazuh Rule 60204 — Level 10 (Brute Force Pattern Detected!)
```

---

### Windows Event IDs Observed:

| Event ID | Name | Count | Severity |
|----------|------|-------|----------|
| **4625** | An account failed to log on | 21+ | 🟡 Medium |
| **4624** | An account was successfully logged on | 0 | — |
| **4740** | A user account was locked out | 0* | 🔴 Critical |

> *4740 was not generated as Windows Account Lockout Policy was set to **"Never"** — this is itself a security misconfiguration finding (documented in Recommendations section).

## 9. SOC Analyst Response — What Should Have Been Done

This section shows how a **real SOC analyst** would respond to these alerts — this is what organizations look for.

### Immediate Response (0-15 minutes):

```
Step 1 — TRIAGE
└── Review Rule 60204 alert (Level 10)
└── Note source IP, target, timestamps
└── Confirm it is NOT authorized activity

Step 2 — CONTAIN
└── Block attacker IP at firewall immediately
└── Isolate victim machine from network if breach suspected

Step 3 — INVESTIGATE
└── Check if any login was SUCCESSFUL (Event ID 4624)
└── Review all events from attacker IP
└── Check for other compromised machines on network
```

### Investigation Queries in Wazuh:

**Check if attacker succeeded:**
```
rule.id: 60122 AND data.win.system.eventID: 4624
```

**Find all activity from attacker IP:**
```
data.win.eventdata.ipAddress: <Kali-IP>
```

**Check for lateral movement:**
```
rule.groups: authentication_success AND agent.name: Window-agent
```

### Short-term Recommendations:

| Finding | Risk | Recommendation |
|---------|------|----------------|
| SMB Port 445 exposed | 🔴 Critical | Block inbound SMB from untrusted networks |
| RDP Port 3389 exposed | 🔴 Critical | Restrict RDP access, enable NLA |
| No Account Lockout Policy | 🔴 Critical | Set lockout after 5 failed attempts |
| SMB Signing Disabled | 🟡 High | Enable SMB signing via Group Policy |
| No network segmentation | 🟡 High | Separate attacker VLAN from victim VLAN |

### Account Lockout Policy Fix (Windows):

```
Group Policy Editor (gpedit.msc):
Computer Configuration
  → Windows Settings
    → Security Settings
      → Account Policies
        → Account Lockout Policy

Set:
  Account lockout threshold    : 5 attempts
  Account lockout duration     : 30 minutes
  Reset lockout counter after  : 30 minutes
```

### Enable Wazuh Active Response (Auto-block):

```xml
<!-- /var/ossec/etc/ossec.conf on Wazuh Manager -->
<active-response>
  <command>firewall-drop</command>
  <location>local</location>
  <rules_id>60204</rules_id>
  <timeout>600</timeout>
</active-response>
```

This automatically blocks the attacking IP for **10 minutes** when Rule 60204 triggers — no manual intervention needed.
